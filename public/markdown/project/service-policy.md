# 📋 서비스 운영 및 비용 최적화

개인 프로젝트라도 실제 서비스처럼 운영한다는 관점에서,  
AI 기능 사용량과 서버 가용성을 함께 고려했습니다.  
특히 OpenAI API 비용이 무제한으로 증가하지 않도록 제한을 두고,  
무료 플랜 환경에서도 사용자가 불편을 느끼지 않도록 운영 방식을 정리했습니다.

---

## 1. AI 사용량 제한

### 사용자별 일일 호출 횟수 제한

OpenAI API 비용을 제어하기 위해 사용자별 일일 사용 횟수를 관리했습니다.

- `AiUsedCount` 테이블을 두어 사용자별 호출 횟수 추적
- 하루 최대 10회까지 사용 가능하도록 제한
- 날짜가 바뀌면 카운트를 초기화해 다시 사용할 수 있도록 구성

```prisma
model AiUsedCount {
  userId    Int      @unique
  user      User     @relation(fields: [userId], references: [id])
  count     Int      @default(0)
  date      String
}
```

**동작 방식**:

- 저장된 날짜가 오늘이 아니면 count를 1로 초기화하고 날짜를 오늘로 갱신
- 오늘 날짜라면 count가 10 미만일 때만 증가
- 제한 횟수를 초과하면 API 호출을 막고 안내 메시지 반환

이를 통해 AI 기능을 무제한으로 열어두지 않고,  
서비스 운영 비용을 통제 가능한 범위 안에서 유지할 수 있도록 했습니다.

### 동일 일기에 대한 중복 호출 방지

같은 일기에 대해 반복적으로 AI 조언을 요청하면  
불필요한 비용이 계속 발생할 수 있기 때문에 중복 호출 방지 장치를 두었습니다.

- `Diary` 테이블의 `hasReceivedAdvice` 플래그로 조언 수신 여부 관리
- 이미 조언을 받은 일기는 버튼을 비활성화해 재요청 방지

```prisma
model Diary {
  // ... 기타 필드
  hasReceivedAdvice Boolean @default(false)

  @@unique([userId, date])
}
```

이렇게 해서 사용자는 동일한 결과를 반복 요청할 수 없도록 했고,  
불필요한 OpenAI 호출도 줄일 수 있었습니다.

---

## 2. 비용 최적화 전략

AI 기능은 단순히 붙이는 것보다 지속 가능하게 운영하는 것이 중요하다고 생각했습니다.  
그래서 다음과 같은 방식으로 비용을 최적화했습니다.

### 경량 모델 사용

`gpt-4o-mini`를 사용해 응답 품질과 비용의 균형을 맞춤

### 응답 길이 제한

응답을 300자 이내로 제한해 토큰 사용량 절감

### 결과 저장

생성된 조언을 DB에 저장해 동일 요청에 대한 재호출 방지

### 최소 호출 원칙

꼭 필요한 시점에만 AI 요청이 발생하도록 설계

즉, AI 기능을 단순 실험성 기능으로 두지 않고  
실제 서비스 환경에서도 운영 가능한 수준으로 다듬는 데 초점을 맞췄습니다.

---

## 3. 백엔드 서버 가용성 유지

백엔드 서버를 무료 플랜 환경에서 운영했기 때문에,  
일정 시간 이상 요청이 없으면 서버가 절전 상태로 전환되는 문제가 있었습니다.  
이 경우 첫 요청 시 응답이 지연되어 사용자 경험이 나빠질 수 있었습니다.

이를 완화하기 위해 외부 헬스체크 서비스를 활용해  
주기적으로 서버에 요청을 보내는 방식으로 가용성을 유지했습니다.

- 5분마다 헬스체크 요청 전송
- 서버가 절전 상태로 내려가지 않도록 유지
- 첫 진입 시 지연되는 문제 완화

이 방식으로 제한된 인프라 환경에서도  
보다 안정적인 사용자 경험을 제공할 수 있도록 운영했습니다.
